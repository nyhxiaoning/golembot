## 结构大纲

### 一、开头：一个真实的痛点场景（300-400 字）

从一个具体的场景切入——

> 你的团队同时在用 Cursor、Claude Code、Codex，每个人偏好不同。
> 有人在飞书群 @bot 问代码问题，有人在 Slack 里让 bot 跑部署脚本，有人直接命令行对话。
> 但底层其实是同一个 AI 助手，同一套 Skills，同一个工作目录。

**核心矛盾**：Coding Agent 的能力越来越强，但每个 Agent 都是孤岛——不同引擎、不同渠道、不同上下文，无法统一管理。

> 这就是我做 GolemBot 的起点。

---

### 二、观察：AI Agent 生态的三个割裂（600-800 字）

**2.1 引擎割裂**

- Cursor、Claude Code、OpenCode、Codex…… 每隔几周就有新的 Coding Agent 发布
- 每个都有自己的 CLI、协议、会话管理方式
- 团队被迫「选边站」，但没有人知道下一个月哪个 Agent 最强
- **产品洞察**：引擎是快速迭代的，绑死一个引擎 = 绑死一个风险

**2.2 渠道割裂**

- 开发者用终端，PM 习惯 Slack/飞书，运维在钉钉
- 同一个 AI 助手的能力，被切成了 N 个互不相通的入口
- **产品洞察**：AI 应该去用户在的地方，而不是让用户来找 AI

**2.3 知识割裂**

- Agent A 在飞书群里听到了一个技术决策，Agent B 在 Slack 里被问到同一个问题时一无所知
- 即使是同一个 Agent，不同渠道的会话彼此隔离
- **产品洞察**：知识应该跟着项目走，不是跟着渠道走

> 一句话总结：**Coding Agent 的能力已经溢出了「终端里对话」这个形态，但基础设施没跟上。**

---

### 三、GolemBot 的产品定位：做三个「不做」（800-1000 字）

> 在 AI 领域，克制比野心更稀缺。

**3.1 不做 Agent 本身**

- GolemBot 不管理上下文窗口、不调度工具、不做推理
- 所有「智能」行为委托给底层的 Coding Agent（Cursor / Claude Code / OpenCode / Codex）
- **为什么**：Agent 能力在以周为单位进化，自己做 Agent = 永远追不上。做「壳」= 自动享受每个引擎的能力升级
- **类比**：GolemBot 之于 Coding Agent，就像 Docker 之于应用 —— 不关心里面跑什么，只提供标准化的运行环境

**3.2 不做复杂抽象**

- 整个框架只有两个核心概念：**Assistant 目录** + **Skill**
- 没有 Pipeline、没有 Registry、没有 Plugin System
- 一个目录就是一个助手，里面放 Skills、放配置、放记忆文件，透明可见
- **为什么**：开发者最讨厌的事情就是「学一个框架比解决问题还难」。5 行 YAML + 一个目录结构，10 分钟上手
- **设计哲学**：目录即助手，文件即状态，git 即版本管理

**3.3 不做封闭平台**

- 引擎可换（今天 Claude Code，明天 Codex，后天还没出生的 Agent）
- 渠道可插拔（6 个内置 IM adapter + 自定义 adapter 接口）
- Skills 是纯 Markdown 文件，不是代码插件，任何人都能写
- **为什么**：AI 生态变化太快，封闭 = 在沙滩上建城堡

> **一句话定位：GolemBot 是 Coding Agent 的「多路复用器」—— 一个助手，多个引擎，多个渠道，一份知识。**

---

### 四、几个关键的产品取舍（600-800 字）

**4.1 Skill 为什么是 Markdown 而不是代码？**

- Coding Agent 本身就能读 Markdown、理解自然语言指令
- 如果 Skill 是 Python/TS 代码，就需要运行时、依赖管理、沙箱……复杂度指数增长
- Markdown Skill = 零依赖 + 人类可读 + AI 可执行
- **代价**：没有传统「插件」那么精确可控。但在 Agent 能力足够强的前提下，自然语言指令的灵活性远超硬编码

**4.2 为什么群聊不只是「在群里 @bot」？**

- 大多数 IM bot 只在被 @mention 时才响应——这是最安全但最笨的模式
- GolemBot 设计了三种群聊策略：
  - `mention-only`：经典模式，被叫到才说话
  - `smart`：bot 持续观察群对话、保持记忆更新，但只在有价值时发言（输出 `[PASS]` 时静默）
  - `always`：每条消息都回，适合专用小群
- **产品思考**：`smart` 模式的本质是让 AI 从「被动工具」变成「团队成员」—— 它一直在听，一直在记，但不是每次都说话

**4.3 为什么选择开源？**

- AI 基础设施层的信任成本极高——企业不会把代码仓库接入一个不透明的 SaaS
- 开源 = 可审计 + 可自部署 + 社区驱动的 adapter/skill 生态
- 商业模式思考（如果适合展开的话）

### 五、一个实际场景：多 bot 群聊协作（400-500 字）

用一个生动的例子收束前面的理论——

> 场景：一个 Slack 群里有两个 GolemBot —— 「研究员」和「代码专家」。
>
> 产品经理在群里说：「@研究员 帮我分析一下用户反馈中最多的三个问题」
>
> 研究员 bot 分析完毕，总结在群里。
>
> 代码专家 bot 一直在「smart 模式」下观察，看到分析结果后自动 @研究员 说：
> 「第二个问题我在代码里见过相关 TODO，要我查一下吗？」
>
> 产品经理说：「@代码专家 查吧」
>
> 代码专家扫描代码库，给出修复建议。
>
> 全程：两个 bot 共享群上下文 + 各自维护记忆文件 + 人类只需要自然语言协调。

> **这不是科幻。这是 GolemBot 今天就能做到的事情。**

### 六、结尾：对 AI Agent 生态的一点思考（300-400 字）

- Agent 能力会越来越强，但**编排 Agent 的基础设施**才是长期价值
- 未来不会只有一个「最好的 Agent」，而是不同场景用不同引擎 —— 就像今天没有人只用一种数据库
- GolemBot 赌的是：**「目录即助手」这个抽象层，会比任何单一 Agent 引擎活得更久**

## 写作建议

1. **开头别上来就介绍产品**——用场景/痛点/观察吸引读者，让他们先产生共鸣
2. **多用类比**——Docker/瑞士军刀/多路复用器，帮助非技术读者建立直觉
3. **少贴代码**——公众号受众不会在手机上看代码块。如果要展示简洁性，最多贴一段 5 行 YAML
4. **结尾要有态度**——作为创始人，你对 Agent 生态的判断是什么？这个判断越清晰、越有争议性，文章传播力越强
